# Impeccable — Design Guidelines

> Persistent design context for the **Remal** rental-management platform.
> Read this before any design or frontend work. Frontend lives in [`rental-platform/`](rental-platform/) (Next.js 14 App Router, TypeScript strict, Tailwind 3.4).

## Design Context

### Scope

This context governs the **three operational portals** — the surfaces used by people who interact with the *system itself*:

- **Admin** — `app/(admin)/` — operator console.
- **Owner** — `app/(owner)/owner/` — property-owner portal (view-only).
- **Client** — `app/account/` + `app/(auth)/` — customer self-service & auth.

> **The public marketing/booking storefront** (`app/(public)/`, `app/page.tsx`, `app/units/`) is **out of scope** here. It keeps its existing warm-terracotta hospitality theme (Georgia serif display, fluid section spacing, GSAP scroll choreography). Do **not** apply portal tokens to it, and do **not** copy storefront marketing patterns into the portals.

> **Start fresh.** The current portal styling is a throwaway mock — it borrows undefined shadcn tokens (`bg-card`, `text-muted-foreground`, `text-card-foreground`) that resolve to nothing, and it inherits the storefront's warm palette and loose spacing, which are wrong for dense operational tools. Treat the existing portal look as an anti-reference, not a starting point.

### Users

Remal is a **commission-based vacation-rental brokerage** for Egyptian North-Coast resorts (Palm Hills, Abraj Al Alamein). The platform connects property owners with clients and runs the whole booking lifecycle. Three audiences use the portals:

- **Operators (Admin portal)** — four roles with scoped permissions:
  - *Super Admin* — full control.
  - *Sales* — lives in the **CRM pipeline** all day; moves leads through 7 stages, takes notes, records deposits. Wants speed, keyboard efficiency, and dense at-a-glance lists.
  - *Finance* — works **transaction logs, payouts, refunds, invoices**. Needs precise numbers, alignment, tabular figures, and trustworthy summaries.
  - *Tech* — units & system settings.
  Context: desktop, long focused sessions, repeated daily workflows, high data volume. Their job is to run the business without errors.
- **Owners (Owner portal)** — property owners, not power users. **View-only**: their units, availability, confirmed bookings, earnings, payout history, reviews. Context: occasional check-ins, often mobile, motivated by *"how much have I earned and when do I get paid?"* Job: confidence that their money and bookings are tracked honestly.
- **Clients (Client account)** — consumers managing their own bookings, reviews, notifications, and profile after booking on the storefront. Context: occasional, mobile-first, low patience. Job: see my trips, pay/track, leave a review.

### Brand Personality

**Trustworthy · Efficient · Quietly warm.** Remal ("رمال" — *sands*) is the calm, organized operating system behind a previously informal brokerage. The portals should feel like a precise financial-grade tool that still carries a thread of the brand's hospitality warmth — competence first, personality second. Emotional goals: **confidence and control** for operators; **transparency and trust** for owners; **clarity and reassurance** for clients. Never flashy, never sterile.

### Aesthetic Direction

**Direction: Neutral-pro + terracotta accent** (think Linear / Stripe). A calm, near-neutral gray workspace where the brand's terracotta appears only on the things you act on — primary buttons, active navigation, key focus states. The gray canvas keeps dense data readable for long sessions; the terracotta keeps the tie to the Remal brand.

- **Theme:** **Light only.** Single source of truth — build with semantic tokens so dark mode *could* be layered later, but do not build or QA a dark theme now.
- **Language / direction:** **English UI now, but RTL-ready.** Build direction-agnostic from day one — use CSS logical properties (`ps/pe`, `ms/me`, `start/end`, `text-start`), never hard-coded `left/right`; keep layouts mirror-safe and avoid directional icons that would break in RTL. Arabic/RTL must be a later *switch*, not a rebuild. (The business is Egyptian; the requirement docs are in Arabic.)
- **Consistency:** **One unified design system, with tonal shifts per portal.** Shared tokens and component library across all three; tune *density and warmth* by audience:
  - **Admin** — densest, most utilitarian. Compact rows, tight rhythm, information-rich.
  - **Owner** — calmer and more spacious; lead with clear money/availability summaries and reassurance.
  - **Client** — warmest and most spacious of the three; the bridge toward the storefront's friendliness (a touch more terracotta, larger imagery), while still part of the system.
- **References (feel):** Linear (crisp, fast, keyboard-friendly chrome), Stripe & Mercury (financial trust, tabular precision, restraint), Notion (calm content density). **Anti-references:** the current mock; the marketing storefront's heavy warm shadows and serif headings; gradient-heavy / glassmorphism "dashboard template" looks; cramped enterprise grids with no whitespace; playful or toy-like UI.

### Design Principles

1. **Data legibility over decoration.** Every screen is mostly numbers, statuses, and lists. Optimize for scannability — strong typographic hierarchy, tabular numerals for money/dates, generous-but-tight alignment, hairline borders over heavy shadows. If an element doesn't aid comprehension or action, remove it.
2. **Terracotta is a spotlight, not a wash.** The neutral gray system carries the UI; `primary` (terracotta `#E87651`) marks the single primary action, active nav, and key focus only. Status meaning comes from a dedicated, accessible status-color system — never from the brand accent.
3. **Status is always unambiguous.** The booking pipeline (prospecting → relevant → no_answer → not_relevant → booked → confirmed → check_in → completed, + cancelled / left_early) and finance states drive the whole app. Encode status with **color + icon + label** together (never color alone), consistently everywhere it appears.
4. **Fit the portal to its user.** One system, three tempos: Admin earns density and speed; Owner and Client earn space, plain-language summaries, and reassurance. Match component density and tone to the audience rather than shipping one flat look.
5. **RTL-ready, accessible, and calm by default.** Direction-agnostic layout (logical properties), **WCAG 2.1 AA** contrast and focus-visible states, ≥40px interactive targets, and functional motion only (120–200ms transitions; respect `prefers-reduced-motion`, already wired in `globals.css`). No scroll theatrics in the portals.

### Foundations (concrete starting tokens)

Define a **portal-specific token layer** distinct from the storefront's warm tokens. Suggested values:

- **Neutral scale (portal canvas/ink):** `50 #F7F8FA` (app canvas) · `100 #EEF0F3` · `200 #E4E7EB` (borders) · `300 #CDD2D9` · `400 #9BA2AD` (placeholder) · `500 #6B7280` (muted text) · `600 #4B5563` · `700 #353A42` (body) · `800 #22262C` · `900 #14171A` (headings/ink). Surfaces/cards `#FFFFFF`.
- **Primary (brand accent):** reuse the existing terracotta scale — `primary-500 #E87651`, `600 #C75D3E` (hover), `700 #A54832` (active), `100/200` for tints. Use sparingly.
- **Status system (AA on white):** success `#15803D` / bg `#DCFCE7`; warning `#B45309` / bg `#FEF3C7`; error `#B91C1C` / bg `#FEE2E2`; info `#1D4ED8` / bg `#DBEAFE`. Build a single source-of-truth map from each pipeline/finance status → {color, bg, icon, label}.
- **Typography:** keep the **offline-safe stack** (no `next/font/google`) — system sans (`Segoe UI`, Arial) for all portal UI; **drop the Georgia serif display** in portals (reserve it for the storefront). Use **`tabular-nums`** for all money, counts, and dates. Tight, functional type scale (≈13–14px base in Admin, ≈14–15px in Owner/Client).
- **Shape & elevation:** crisper than the storefront — `~8px` card radius, `~6px` controls; lean on **1px neutral-200 borders + subtle cool shadow**, not the warm heavy storefront shadows. 8px spacing grid.
- **Reuse what works:** keep the existing component API conventions (`components/ui/*` — `Button` variants/sizes, `DataTable`, `StatusBadge`, `Drawer`, `Modal`, `Skeleton*`), TanStack Query/Table, Recharts, lucide-react icons, and sonner toasts — restyle them onto the new token layer rather than reinventing the contracts.
